# LIFE_TRACK 재미 강화 방향성 — 설계 검토 요청

> GPT / Gemini / Cursor 대상 교차 검토용 문서. 콜드 리더 기준으로 작성됨.
> 작성 시점: 2026-04-22

---

## 0. 이 문서는 무엇인가

한국 학창시절 7년(초6~고3)을 주차 단위로 걷는 서사 라이프시뮬 `LIFE_TRACK: 선택의 결과`에 대한 재미 강화 방향성 초안. 현재 개발자는 시스템 상당 부분이 "의례화(ceremonial)"됐다고 진단했고, 개편 방향 5가지를 도출했다. 본 문서는 그 도출 과정과 결론을 정리한 것이며, 외부 AI 3곳(GPT, Gemini, Cursor)에 교차 검토를 요청하기 위해 작성됐다.

## 1. 프로젝트 스냅샷

### 1.1 개요
- 장르: 주차 단위 진행 라이프시뮬 (Persona / Princess Maker 계통)
- 시대/정서: 2012~2015 한국 학창시절 노스탤지어
- 기술: React 19 + Vite + Zustand + TypeScript
- 플레이 길이: 7학년 × 48주 = 336주

### 1.2 핵심 시스템
- **스탯**: academic / social / talent / mental / health (0~100)
- **자원**: 피로(0~100), 용돈(만원 단위)
- **루틴**: 주중 방과후 슬롯 2개 (self-study, 운동, 학원, 노는 것 등)
- **주말/방학**: 별도 선택 슬롯
- **이벤트**: 고정(생일·졸업·시험) + 조건부(친밀도·스탯) + 랜덤 풀
- **NPC 6명**: 지훈(소꿉친구), 민재(공부라이벌), 유나(도서관), 수빈(학원), 하은(선배멘토), 준하(전학생)
- **상점 18개 아이템**: 편의점/서점/선물/패션/특별 5카테고리
- **부모 특성 2개**: wealth/info/gene/emotional/freedom/strict 중 선택 → 초기 유리점
- **엔딩**: Y7 말 스탯·관계·선택 기반

### 1.3 콘텐츠 밀도
- 이벤트 ~150+, NPC 전용 체인(첫만남→친밀→위기→약속) 풍부
- 시험 시스템 완비 (과목별, 석차, 부모/교사 반응)
- Y1(초6) CG 에셋 대량 확보 중

## 2. 진단 데이터 (시뮬 기반)

개발 전에 "상점이 의사결정 피로를 유발한다"는 가설로 2회 서브에이전트 진단을 돌렸다. 결과는 예상 밖이었다.

### 2.1 상점 피로 진단
- 주당 평균 **14개** 아이템이 구매 가능 상태
- **100% 주차**에서 "2개 이상 고려 상황" 발생
- 매주 기계 구매(greedy) vs 전혀 미구매(none) 최종 스탯 격차 **+15.6%**뿐
- 구매 전혀 안 하면 Y7 말 잔고 **1,540만원** 잉여 → 돈 싱크 실패
- **데드 아이템 13/18 (72%)** — greedy 전략조차 고가·조건부는 안 삼

### 2.2 상점 의존도 진단 (후속)
- 피로 자연 회복 `max(3, min(12, fatigue × 0.15))` + 추가 보정 → **자기조절형**, 평형점 ~50
- 합리적 플레이에서 피로 80+ **0회**, 번아웃 **0회**
- "쉬기 완전 무시"하는 극한 플레이에서도 피로 최고 77.8, 번아웃 0회
- 현실적 유저 모델(피로≥60 시만 구매)은 baseline에서 **구매 트리거 0회 = 지출 0만원**

**결론**: 이전 가설("자연 회복 부족해서 소모품 의존")은 데이터상 **틀렸다**. 자연 회복은 오히려 **과잉**이다. 소모품은 필수도 치료제도 아니며, greedy 시뮬의 아티팩트였을 뿐.

## 3. 개발자의 재해석

진단 데이터가 드러낸 것은 단순히 "상점이 약하다"가 아니라 **시스템 전반의 의례화**다.

### 3.1 재미의 6축 현황

| 축 | 상태 | 문제 |
|---|---|---|
| Agency (내 선택이 중요) | 이벤트 레벨 OK, 메타 레벨 약 | 스탯·돈의 실질 영향 불투명 |
| Surprise (예상 못한 순간) | 스케줄 이벤트 풍부 | 진짜 우연/긴급 부재 |
| Narrative (캐릭터 애정) | NPC 체인 훌륭 | NPC 간 상호 관계 없음 (사일로) |
| Progress (성장감) | 스탯 올라감 | 엔딩/관계 반영도 불명확 |
| Tension (실패 가능성) | 사실상 없음 | 번아웃 0회, 위기 0회 |
| Exploration (다른 길) | 스케줄 이벤트 많음 | "다시 플레이할 동기"가 약 |

### 3.2 약점 Top 3

1. **"선택의 결과"가 결과를 안 보여준다** — Y7 말 엔딩이 스탯 합산에 가까우면 중간 3,000개의 선택은 "수치 변동"으로만 흡수. 이벤트 대사는 살아있지만 이벤트끼리 참조·인과가 약함.

2. **NPC가 친구 5명이 아니라 "친구 5개의 독립 앱"** — 각 NPC 체인은 훌륭. 그런데 민재가 유나를 언급하는 장면? 수빈·하은의 학원 인연? 없음. "반이 살아있다"는 감각이 없음.

3. **위기가 없다 = 서사의 드라마도 약해진다** — 번아웃·실패·갈등 구간이 극히 드물면 플레이는 "잔잔한 산책". 학창시절 회고록이라면 드라마틱 구간이 정서의 기둥.

## 4. 5가지 방향 제안

### 방향 A — "회고록 해상도": 엔딩·서사 중심
- 엔딩을 "스탯 합산" → "결정적 순간 N개 + 핵심 관계 + 진로"로 재설계
- 엔딩 텍스트가 플레이어의 **실제 선택을 인용**
- "초6 W5 때 너는 □□을 선택했다" 회상 시퀀스

### 방향 B — "관계의 그물": NPC 상호작용
- NPC 간 배경 관계 그래프 도입 (우정/경쟁/갈등)
- 유나 친밀도 올리면 민재가 "쟤 공부 잘한다던데" 언급
- 2명 이상 그룹 이벤트 확대

### 방향 C — "드라마 구간" 강화
- 첫 번아웃·첫 시험 실패·첫 관계 위기를 **필연**으로 배치
- 극복 이벤트 체인이 엔딩 톤 전환
- 잔잔한 회고록 → 성장 드라마로 쉬프트

### 방향 D — "메커니즘 정직하게"
- 회복 과잉 조정, 돈 싱크 추가, 이벤트 조건에 스탯 영향 강화
- 관대한 톤 잃음, 메커니즘 의미 획득

### 방향 E — "재플레이 동기"
- 엔딩 트로피/앨범, 놓친 루트 힌트
- 성별·부모·진로별 차별화된 경험

## 5. 개발자 추천: A + B 조합

### 이유
- 게임 **정체성**과 가장 부합 ("한국 학창시절 잔잔한 회고록")
- 기존 자산(이벤트 체인, NPC 설계)을 **레버리지**하는 방향 — 새 시스템 추가가 아닌 기존 연결
- C·D는 **톤 변경 위험**
- E는 A·B로 일부 자동 해결

### 안 하는 게 나은 것
- 상점 전면 개편 (비중 대비 투자 과함)
- 자연 회복 재조정 (톤 훼손)
- 복잡한 새 시스템 추가 (이미 복잡)

### 구체적 첫 스텝
1. 엔딩 시스템 점검 (1~2일) — 현재 분기 규칙, 스탯 의존도, 부족 축 정리
2. NPC 관계 그래프 도입 (2~3일) — 각 NPC에 "다른 NPC와의 관계" 배경 + 이벤트 대사 2차 언급
3. 엔딩 텍스트 서사화 (2~3일) — 7년 회고 시퀀스, 실제 선택 인용

## 6. 검토 요청 사항

아래 질문에 답해주길 바란다. 반대 의견·놓친 관점·대안 조합 환영.

1. **진단 진단**: 본 문서가 그린 현 시스템 문제(의례화·NPC 사일로·위기 부재)가 타당한가? 과잉 해석이거나 놓친 부분?
2. **방향성 평가**: 5가지 방향 A~E 중 어느 조합이 **이 게임 정체성**(한국 학창시절 회고록)에 최적인가?
3. **추천 검증**: 개발자 추천 "A + B"에 동의하는가? 다른 조합 추천?
4. **우선순위 첫 스텝**: 첫 2주간 어디에 집중해야 하나? 제안된 3스텝이 적절한지, 혹은 다른 출발점 추천?
5. **리스크**: 이 방향으로 갈 때 예상되는 함정·실패 패턴은?
6. **참고 사례**: 유사 장르에서 같은 방향으로 성공/실패한 게임 사례?

## 부록: 기술적 맥락

- 이벤트는 단일 `events.ts` 파일에 배열로 정의. `condition: (state) => boolean` 기반 필터링
- NPC는 `types.ts`의 `NpcState`로 `{id, name, intimacy, met}` 보유 — 관계 그래프 필드 없음
- 엔딩은 `calculateEnding()` 함수에서 스탯·career 기반 계산 (개발자 아직 전부 못 봄)
- 코드베이스는 정돈돼 있으나 NPC 간 참조는 거의 없음
