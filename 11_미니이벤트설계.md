# 11. 미니 이벤트 설계 (말걸기 시스템)

## 설계 철학

> "메인 이벤트가 각본이라면, 미니 이벤트는 일상의 결이다."

- 메인 이벤트는 **NPC가 변하는 순간**, 미니 이벤트는 **NPC가 살아있다는 증거**
- 친밀도는 숫자가 아니라 **그 NPC와 함께 본 장면의 누적**이다
- 풀이 빈약하면 캐릭터가 빈약해진다 — NPC당 1개뿐이면 그 캐릭터는 7년 내내 같은 사람
- 미니 이벤트는 **선택지가 없다** — "지금 이 NPC가 너에게 보여준 한 장면"을 받아들이는 것
- 사전 결정 모델 — 클릭은 굴리는 행위가 아니라 이미 정해진 결과를 드러내는 행위

---

## 1. 시스템 개요

### 1-1. 메인 이벤트 vs 미니 이벤트

| 구분 | 메인 이벤트 | 미니 이벤트 |
|------|-------------|-------------|
| 발동 | 주차/조건 매칭 시 자동 | 플레이어가 "말 걸기" 클릭 시 |
| 빈도 | 7년 내 ~270개, 점진 등장 | 매주 0~1회 (pressure 기반 사전 결정) |
| 본문 | 지문 + 선택지 2~4개 | 지문 + 효과 한 줄 (선택지 없음) |
| 효과 | 스탯/관계 큰 폭, 메모리 슬롯 | 스탯/관계 소폭, 메모리 슬롯 드물게 |
| 재발동 | 일부 ANNUAL | 1회 발동 후 영구 풀 제외 |
| 위치 | events.ts | talkSystem.ts |
| CG | 일부 보유 | 없음 (텍스트 only) |

### 1-2. Pressure 메커니즘 (사전 결정 모델)

```
매주 시작 (processWeek):
  talkEventPressure += 0.1   // 0~1로 클램프
  npcEventPendingThisWeek = (seededRandom() < talkEventPressure)
  parentEventPendingThisWeek = (... + 0.05 누적)

플레이어가 "말 걸기" 클릭 (talkToNpc):
  if (친밀도 ≥ 30 && pending && 그 NPC 풀에 fired 안 된 이벤트 있음):
    fire → 효과 적용, pressure 0 리셋, pending 소비, fired 기록
  else if (pending=true이지만 그 NPC 풀이 비었음):
    잡담 한 줄 (pending 보존 → 다음 NPC가 가져감)
  else:
    잡담 한 줄 (pressure는 매주 +0.1 누적되며 차오름)
```

핵심:
- **"이번 주에 이벤트가 발동될지"는 사전 결정**
- **"누구에게 갈지"는 클릭 순서에 따름**
- 풀 빈 NPC는 pressure 누수 통로가 됨 (의도된 동작이지만 풀 부족하면 체감 악화)

### 1-3. Smalltalk vs 미니 이벤트 (플레이어가 보는 차이)

| | Smalltalk (잡담) | 미니 이벤트 |
|---|---|---|
| 효과 | 없음 | 친밀도 +2~5, 스탯 ±1~2 |
| 빈도 | 클릭마다 (재사용 무한) | NPC당 평생 1회 |
| 길이 | 한 줄 | 짧은 장면 + 효과 |
| 톤 | "오늘 어땠어?" 류 일상 | "이번 주의 한 장면" |
| 풀 위치 | NPC_SMALLTALK | NPC_MINI_EVENTS |
| 풀 크기 (현재) | NPC당 8~12개 | NPC당 1개 (← 본 문서가 해결하려는 문제) |

플레이어 체감:
> "친밀도 30 넘긴 다음 처음 말 걸었을 때, 평소랑 다른 한 장면이 나오면 → 미니 이벤트.
> 그 장면을 본 후엔 한동안 다시 잡담만 → pressure 차오르며 다음 미니 이벤트 대기."

### 1-4. 가정(부모) 미니 이벤트

NPC 미니 이벤트와 동일 메커니즘. 부모는 단일 엔티티로 두 강점 풀의 합집합.
- pressure 누적률: NPC +0.1, 부모 +0.05 (덜 자주)
- 부모 풀 (현재): 강점당 1개씩 6개

## 2. 풀 설계 원칙

### 2-1. NPC당 풀 분포 — 친밀도 단계별 잠금 해제

> 한 캐릭터를 7년간 따라가게 하려면 **친밀도 구간마다 결이 다른 미니 이벤트**가 필요하다.
> "처음 친해진 상대"와 "오래 친한 상대"는 보여주는 결이 다르다.

| 친밀도 단계 | 톤 | 한 줄 정의 |
|---|---|---|
| **30+ (가벼운 일상)** | 같이 시간 보내는 사이 | "같이 뭐 할래?" — 활동 제안 |
| **50+ (일상 코드)** | 두 사람만의 약속·습관 | "그거 또 시작이네" — 서로의 결을 알아본 사이 |
| **70+ (속마음)** | 평소 안 보이던 약한 면 | "...너한테만 말하는 건데" — 무리에선 안 꺼내는 얘기 |
| **90+ (코어)** | 캐릭터 결정적 한 장면 | 그 NPC를 "그 NPC답게" 만드는 사건. 메모리 슬롯 필수. |

권장 분포: **NPC당 4~6개** (단계마다 1개 이상).

### 2-2. 효과 가이드라인

| 친밀도 단계 | 친밀도 변동 | 스탯 변동 | 피로/돈 | 메모리 슬롯 |
|---|---|---|---|---|
| 30+ | +2 | ±0~1 (단일) | 0~±1 | X |
| 50+ | +3 | ±1 (단일~2개) | ±1 | X |
| 70+ | +4 | mental ±2 (감정형) | -1 ~ +1 | importance 3 후보 |
| 90+ | +5 | 카테고리 의미 | -2 ~ +2 | importance 5 (필수) |

원칙:
- 친밀도가 깊어질수록 효과가 커지지만, **수치 자체보다 톤이 우선**
- 90+ 미니 이벤트는 메모리 슬롯 importance 5로 엔딩 회상에 등장 가능
- 70+ 미니 이벤트는 카테고리 다양화 (discovery/reconciliation/growth) 선호

### 2-3. 학년/시기 게이팅

- **NPC 등장 학년 이전**: 풀 자체 진입 불가 (intimacy 게이트와 별개로 학년 가드)
- **초등/중등/고등 톤 분리**:
  - 초등: 단순한 놀이, 분식, 학원
  - 중등: SNS, 친구 그룹, 첫 진로 고민의 씨앗
  - 고등: 야자, 모의고사, 진학 압박, 진지한 대화 톤
- **시즌성 (선택적)**:
  - 봄: 벚꽃, 신학기 어색함
  - 여름: 매점 빙수, 방학 직전 들뜸
  - 가을: 체육대회 직후 피로
  - 겨울: 핫팩, 시험 후 카페

학년·시즌 게이팅은 "풀이 진짜로 살아있다"는 체감의 핵심. Phase 2.5에서 본격 도입.

### 2-4. 성별 분기 정책

events.ts의 `femaleDescription/femaleChoices` 패턴과 결을 일치시킨다.

| 분기 필요 시점 | 처리 |
|---|---|
| 톤이 성별로 명확히 갈리는 활동 | `gender: 'male' | 'female'` 두 시드로 분리 (예: 지훈 농구 ↔ 배드민턴) |
| 톤이 성별 무관 | 단일 시드, gender 필드 생략 (대부분의 케이스) |
| 같은 활동인데 NPC 대사 톤만 미묘하게 다름 | 단일 시드 유지, smalltalk common+gender 풀로 변주 흡수 |

원칙: **성별 분기는 비용**. 이유 없이 분기하면 풀이 두 배로 늘고 작가 부담 증가.

### 2-5. 메모리 슬롯 후보 정책

미니 이벤트는 기본적으로 메모리 슬롯을 만들지 **않는다** (메인 이벤트 우선).

예외:
- **친밀도 70+ 미니 이벤트** 중 톤이 강한 것 → importance 3 슬롯 후보
- **친밀도 90+ 미니 이벤트** 전부 → importance 5 슬롯 (필수, recallText 작성 필요)
- 카테고리는 NPC 아크에 맞춰: jihun=courage/growth, subin=discovery, minjae=growth, yuna=discovery, haeun=reconciliation, junha=warmth(=discovery)

> 7년에 NPC 6명 × 90+ 미니 이벤트 1개씩 = **6개의 "그 NPC다운" 결정적 장면**이 엔딩 회상에 들어갈 수 있어야 한다.

- Phase 2.6 (선택적): 강점당 30/70 두 단계만 — 강점당 +1, 총 +6개

## 4. 풀 소진 후 처리

### 4-1. 현재 동작 (Phase 2.1)

- 풀 다 본 NPC → `talkEventsFired`에 영구 기록, 매주 잡담만 노출
- pressure는 매주 +0.1 누적되며 다른 NPC 풀에서 발동
- **체감**: "이 NPC는 더 새로운 게 없네" → 1회성 이벤트 NPC로 인식됨

### 4-2. 옵션 A — 잡담 풀 보강만 (최소 비용)

- NPC당 잡담 풀이 충분(8~12개)하면 OK
- 미니 이벤트가 끝난 NPC도 친밀도는 메인 이벤트로 유지
- 단점: 풀 소진 후엔 "관계가 정체된 느낌"

### 4-3. 옵션 B — 친밀도 단계별 자동 해제 (**채택**)

본 문서의 NPC당 4단계 풀 설계가 그대로 옵션 B의 구현이다.
- intimacy 30 미니 이벤트 fired 후에도, intimacy 50 도달 시 새 풀 1개 열림
- 단계별 풀이 자연스럽게 다음 만남을 약속
- 작업량: NPC당 +3 = 6 NPC × 3 = **18개 시드** (Phase 2.2~2.4 분할)

### 4-4. 옵션 C — fired 만료 / 재발동 (비추)

- 30주 지나면 talkEventsFired에서 ID 제거 → 같은 미니 이벤트 재발동
- 단점: 변별력 떨어짐, 메모리 슬롯이 중복 생성될 위험
- **채택 안 함**

## 5. Phase 분할

### Phase 2.1 (✅ 완료)
- 시드 풀: NPC당 1개 (intimacy 30+)

- 메모리 슬롯 없음
- 효과: 친밀도 +3, 스탯 ±1
- 작가 가이드: NPC 캐릭터 프로필 다시 읽고 "그 NPC만의 습관·취향"이 드러나는 한 장면

### Phase 2.3 — intimacy 70 풀 (속마음)
- NPC 6명 × 1개 = **6개 시드** 추가
- 톤: "평소 안 보이는 약한 면"
- 일부 메모리 슬롯 importance 3 후보
- 효과: 친밀도 +4, mental ±2
- 작가 가이드: 그 NPC의 "공식 페르소나"가 살짝 깨지는 한순간

### Phase 2.4 — intimacy 90 풀 (코어)
- NPC 6명 × 1개 = **6개 시드** 추가
- 톤: "그 NPC를 그 NPC답게 만드는 결정적 장면"
- **모두 메모리 슬롯 importance 5** (recallText 필수)
- 효과: 친밀도 +5, 카테고리 의미 부여
- 작가 가이드: 7년 끝에 회상으로 떠오를 한 장면. 가장 신중하게 작성.

### Phase 2.5 — 학년/시즌 게이팅
- 기존 시드에 학년 condition 부여 (예: jihun_50_pcbang은 Y3+)
- 신규 시즌 이벤트 (여름 빙수, 겨울 핫팩 등) 4~6개
- 작업량 TBD

### Phase 2.6 (선택적) — 부모 풀 단계 분기

## 6. 시드 작성 템플릿

```typescript
{
  id: 'talk_<npcId>_<intimacyTier>_<keyword>',
  // 예: talk_jihun_50_pcbang, talk_subin_70_alone

  npcId: '<npcId>',
  intimacyMin: 30 | 50 | 70 | 90,
  gender?: 'male' | 'female',     // 톤 분기 필요시만
  yearMin?: number,               // 학년 게이팅 (예: 고등 톤이면 5)
  yearMax?: number,               // 도윤처럼 Y1 종료면 1

  description: '<NPC 대사 + 상황 1~3줄>',
  effects: {
    intimacy: 2 | 3 | 4 | 5,         // 단계별 가이드라인
    stats?: Partial<Stats>,
    fatigue?: ±1~3,
    money?: ±1~2,
  },
  message: '<효과 한 줄 요약, UI 노출>',

  // intimacy 70+: memorySlotDraft 가능
  // intimacy 90+: memorySlotDraft 필수
  memorySlotDraft?: {
    category: '<NPC 아크에 맞는 카테고리>',
    importance: 3 | 5,
    toneTag: '<warm | resolve | regret | quiet | ...>',
    recallText: '<엔딩 회상에 노출될 한 줄>',
    npcIds: ['<npcId>'],
  },
}
```

### 작성 체크리스트

- [ ] **그 NPC만 할 수 있는 말인가?** (다른 NPC로 바꿔도 통하면 캐릭터 빈약 — 다시 쓸 것)
- [ ] **친밀도 단계와 톤이 맞는가?** (30 단계인데 너무 깊거나 / 90 단계인데 너무 가볍지 않은지)
- [ ] **학년 의존 표현이 있으면 yearMin/yearMax 또는 condition 추가**
- [ ] **효과치가 가이드라인 안에 있는가** (§2-2 표 참조)
- [ ] **80자 이내로 한 화면에 들어가는가** (description은 1~3줄, 너무 길면 호흡 X)
- [ ] **메모리 슬롯 후보면 recallText까지 작성**
- [ ] **다른 시드와 ID 중복 없는가** (grep으로 확인)
- [ ] **smalltalk common 풀과 톤이 충돌하지 않는가** (잡담은 일상, 미니 이벤트는 한 장면)
